# Cafe24 OAuth 토큰 관리 가이드

최종 업데이트: 2026-03-25

---

## 1. 핵심 정책 (Cafe24 공식)

| 항목 | 내용 |
|---|---|
| access_token 유효시간 | 약 2시간 |
| refresh_token 유효시간 | 최대 14일 (고정, 연장 불가) |
| refresh 요청 결과 | 새 access_token + 새 refresh_token 세트 반환 |
| 기존 refresh_token | 즉시 폐기 (교체 모델, 재사용 불가) |
| refresh_token 만료 시 | authorization_code 재발급 필요 (자동 복구 불가) |

---

## 2. 단순화 배경 (2026-03 작업)

### 문제: 갱신 주체 분산 → Race Condition 위험

단순화 이전에는 3곳에서 동시에 토큰 갱신 가능:

| 주체 | 방식 | 문제 |
|---|---|---|
| monitoring-gas `refreshCafe24Token()` | 1시간 고정 트리거 | - |
| p7_gas `autoRefreshCafe24Token()` | 1시간 고정 트리거 | 동일 refresh_token 동시 소비 가능 |
| p7_gas `buildCafe24Cache()` | 내부 직접 호출 | 동일 refresh_token 동시 소비 가능 |
| Vercel `check-token-expiry` cron | Vercel cron (미등록) | 불필요한 갱신 로직 보유 |

Cafe24의 교체 모델 특성상 두 주체가 동시에 동일 refresh_token으로 갱신을 시도하면 하나는 `invalid_grant` 실패. 결과적으로 새 token 세트가 서로 다른 저장소에 분리 저장되어 불일치 발생.

### 해결: 단일 갱신 주체 (monitoring-gas)

**삭제된 것:**

1. `p7_gas.js` — `autoRefreshCafe24Token()` 트리거 등록 코드 (`createTrigger()` 내부)
2. `p7_gas.js` — `buildCafe24Cache()` 내부 `autoRefreshCafe24Token()` 직접 호출
3. `vercel.json` — `check-token-expiry` cron 항목
4. `src/app/api/cron/check-token-expiry/route.ts` — access_token 자동 갱신 블록

**유지된 것:**

1. `monitoring-gas/Code.js` — `refreshCafe24Token()` 1시간 트리거 (갱신 단일 주체)
2. `p7_gas.js` — `ensureTokenRefreshIfNeeded_()` (만료 2시간 이내 폴백)
3. `p7_gas.js` — `tryRefreshAndRetry_()` (401 감지 시 긴급 폴백)
4. `src/app/api/cron/check-token-expiry/route.ts` — refresh_token 만료 알림 로직

## 3. 토큰 저장 구조

### Redis (메인 저장소)

Key: `cafe24_tokens`

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "expires_at": 1700000000000,
  "refresh_token_expires_at": 1701200000000
}
```

- `expires_at`: Unix ms (now + 2시간)
- `refresh_token_expires_at`: Unix ms (카페24 응답의 ISO 문자열 파싱)

### Vercel 환경변수 (Redis 유실 대비 fallback)

- `CAFE24_ACCESS_TOKEN`
- `CAFE24_REFRESH_TOKEN`
- monitoring-gas가 갱신 시 `/api/token-update`로 동기화

### 구글 시트 [설정] (GAS용 저장소)

- 모니터링 시트 [설정]: monitoring-gas가 직접 읽고 씀
- 매핑테이블 시트 [설정]: p7_gas가 `initMonitoringSheet_()`로 모니터링 시트에서 오버레이

## 4. 현재 토큰 갱신 구조

### 단일 갱신 주체: monitoring-gas

```
monitoring-gas (1시간 고정 트리거)
  refreshCafe24Token()
    → 모니터링 시트 [설정]에서 refresh_token 로드
    → 카페24 API 호출 (refresh_token grant)
    → 새 token 세트 수신
    ├─ 모니터링 시트 [설정] 업데이트
    └─ POST https://web-cadalog-ver10.vercel.app/api/token-update
         → Redis 저장
         → Vercel 환경변수 업데이트
```

### p7_gas (읽기 전용 + 폴백)

```
각 함수 실행 시작
  → initMonitoringSheet_() → 모니터링 시트에서 최신 토큰 오버레이
  → G_TOKEN 설정

폴백 1 — ensureTokenRefreshIfNeeded_()
  조건: access_token 만료 2시간 이내
  → 직접 갱신 후 모니터링 시트 + 매핑테이블 시트 저장

폴백 2 — tryRefreshAndRetry_()
  조건: 카페24 API 401 응답 감지
  1차: 모니터링 시트 재읽기 (monitoring-gas가 이미 갱신했을 수 있음)
  2차: 직접 refresh_token으로 갱신
```

### Vercel check-token-expiry (알림 전용)

- `/api/cron/check-token-expiry`: 갱신 로직 없음, 알림 전용
- vercel.json cron 미등록 → 현재 자동 실행 안 됨 (수동 호출만 가능)
- refresh_token 만료 D-1 이하: 이메일 알림 발송

## 5. 갱신 주기 정리

| 주체 | 주기 | 역할 |
|---|---|---|
| monitoring-gas `refreshCafe24Token()` | 1시간 고정 | **토큰 갱신 단일 주체** |
| p7_gas `ensureTokenRefreshIfNeeded_()` | 각 함수 실행 시 | 만료 2시간 이내 폴백 갱신 |
| p7_gas `tryRefreshAndRetry_()` | 401 감지 시 | 긴급 폴백 갱신 |
| Vercel `check-token-expiry` | 미예약 (수동) | refresh_token 만료 알림 |

## 6. Refresh 요청 방식 (카페24 공식 API)

```
POST https://{mall_id}.cafe24api.com/api/v2/oauth/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token={현재_refresh_token}
```

응답:
```json
{
  "access_token": "새토큰",
  "refresh_token": "새리프레시토큰",
  "expires_at": "2026-03-25T12:00:00.000",
  "refresh_token_expires_at": "2026-04-08T12:00:00.000"
}
```

## 9. 에러 코드 대응

| 에러 | 원인 | 대응 |
|---|---|---|
| `401 Unauthorized` | access_token 만료 | 자동 갱신 (tryRefreshAndRetry_) |
| `invalid_grant` | refresh_token 만료 또는 재사용 | 재인증 절차 (8번 섹션) |
| `invalid_client` | CLIENT_ID/SECRET 오류 | 환경변수 확인 |

## 10. 운영 요약

| 항목 | 자동 | 수동 |
|---|---|---|
| access_token 갱신 | ✅ monitoring-gas 1시간마다 | - |
| 만료 임박 알림 | ✅ monitoring-gas D-7, D-1 | - |
| refresh_token 갱신 | ❌ (카페24 정책상 자동 불가) | 14일마다 1회 재인증 |

결론: Cafe24 정책상 완전 자동 영구 유지 불가. 현실적 구조 = access_token 자동 갱신 + 만료 전 알림 + 14일마다 1회 재인증.
