# Cafe24 OAuth 토큰 관리 가이드 (공식 정책 정합)

작성 기준: Cafe24 공식 OAuth retoken 정책

---

## 1. 핵심 정책 요약

1. access_token은 약 2시간 유효
2. refresh_token은 최대 14일 유효 (고정 만료)
3. refresh 요청 시 새 access_token + refresh_token 세트가 반환됨
4. 기존 refresh_token은 즉시 폐기됨 (교체 모델)
5. refresh_token 만료 시 authorization_code 재발급 필요
6. refresh_token 수명은 API 호출로 연장되지 않음

---

## 2. 환경 변수 원칙 (.env)

.env에는 토큰을 저장하지 않는다.

```
CAFE24_MALL_ID=daesan3833
CAFE24_CLIENT_ID=...
CAFE24_CLIENT_SECRET=...
CAFE24_REDIRECT_URI=https://your-domain.com/api/auth/callback
KV_REDIS_URL=redis://...
```

금지:
- access_token 저장 금지
- refresh_token 저장 금지

---

## 3. Redis 저장 구조

Key: `cafe24:oauth:{mall_id}`

```json
{
  "access_token": "...",
  "expires_at": 1700000000000,
  "refresh_token": "...",
  "refresh_token_expires_at": 1701200000000
}
```

필드 의미:
- `expires_at` = now + `expires_in`
- `refresh_token_expires_at` = now + `refresh_token_expires_in`

---

## 4. 토큰 동작 흐름

API 요청 시:
1. Redis에서 토큰 로드
2. access_token 만료 여부 확인
3. 만료 전이면 그대로 사용
4. 만료 시 refresh_token으로 재발급 요청
5. 새 토큰 세트 Redis 저장
6. 원래 요청 재시도

---

## 5. Refresh 요청 방식 (공식)

```
POST /api/v2/oauth/token
Headers: Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded
Body: grant_type=refresh_token&refresh_token=...
```

---

## 6. refresh_token 만료 대응

refresh_token은 14일 고정 만료.

만료 시:
- `invalid_grant` 반환
- 자동 복구 불가
- authorization_code 재발급 필요

---

## 7. 만료 알림 전략

Cron (하루 1회 실행):
1. Redis의 `refresh_token_expires_at` 조회
2. D-3 경고
3. D-1 재경고
4. 만료 시 `NEEDS_REAUTH` 상태 기록

---

## 8. 재인증 프로세스

### 8.1 /api/auth/cafe24/start
Cafe24 authorize URL로 redirect

### 8.2 /api/auth/cafe24/callback
code 수신 → 즉시 token 교환 (code는 1분 유효) → Redis 저장

---

## 9. 에러 처리 전략

- access_token 만료: 자동 refresh
- refresh_token 만료: `NEEDS_REAUTH` 기록 → 관리자 재인증 요청

---

## 10. 운영 모델 요약

항목 | 자동 | 수동
---|---|---
access_token 갱신 | ✅ | 없음
refresh_token 갱신 | ❌ | 14일마다 1회

결론:
- Cafe24 정책상 완전 자동 영구 유지 구조는 불가능
- 현실적 구조: access_token 자동 갱신 + refresh_token 만료 전 알림 + 2주마다 1회 재인증
